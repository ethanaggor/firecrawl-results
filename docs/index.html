MCP search upgrade prototype

Make Search queryable for agents

Firecrawl already has the primitives: search, scrape, clean markdown, summaries, and source controls. This prototype wraps those primitives in a bounded context contract that agents can use predictably.

Firecrawl Search API categories country timeout ignoreInvalidURLs →
results summary chunks markdown

The gap is not raw capability.

The gap is the agent-facing contract. Search can already retrieve content, but the MCP surface needs to make context depth, budget, freshness, source policy, and usage metadata explicit.

Before

Agents choose between shallow Search results or a broad Scrape-shaped options blob.

After

Agents ask for results, summaries, chunks, or capped markdown with hard character budgets.

Why It Matters

Firecrawl becomes easier to benchmark against agent search products on context usefulness.

One upgraded tool.

The prototype upgrades firecrawl_search in place. It does not add overlapping search tools. The response stays flattened, compact, and measurable.

Live validation found one docs/API mismatch: /v2/search rejects scrapeOptions.zeroDataRetention. The prototype therefore preserves Search ZDR through enterprise and does not forward unsupported scrape ZDR fields from Search.

Separate parity from product.

The upstream issues are deliberately small: SDK request parity, SDK response metadata access, and MCP Search API parity. The agent-native context contract remains in this prototype.

The benchmark path.

The tracked eval runner starts with contract smoke checks and source-policy experiments. The next layer compares answer quality and context size against Exa, Tavily, and Parallel on shared agent tasks.

Layer 1

Layer 2
