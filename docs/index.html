A context contract for Firecrawl Search

I upgraded firecrawl_search so agents can request bounded web evidence directly instead of choosing between shallow results and raw scrape options.

EA Ethan Aggor - April 24, 2026
Table of Contents tl;dr What changed Why search owns it Why MCP owns it Executor Implementation Parity issues Eval plan
I changed firecrawl_search in place. The new shape adds content, focus, hard character budgets, freshness, source policy, advanced options, and response accounting.
The old MCP shape exposed the primitives, but left the agent to decide too much about scraping, budgeting, and returned context.
The benchmark should compare the previous MCP, the upgraded MCP, Firecrawl content modes, Exa, and Parallel on the same agent tasks.

What changed

The old search tool had an awkward middle ground. Agents could ask for a result list, or they could pass broad scrape options and hope they asked for the right thing. That is not a search contract. That is the agent reaching through Search into Scrape.

MCP search contract old shape and upgraded shape

Before

The agent knew Search existed, then had to infer how much Scrape to pull into it.

firecrawl_search({
  query,
  limit,
  tbs,
  sources,
  scrapeOptions
})
Content depth is implicit.
Budgets are not part of the contract.
Search UX depends on Scrape knowledge.

After

The agent asks Search for a bounded amount of evidence directly.

firecrawl_search({
  query,
  content: "chunks",
  focus,
  maxResults,
  maxCharsPerResult,
  maxCharsTotal,
  freshness,
  sourcePolicy,
  enterprise
})
Content mode is explicit.
Total returned chars are capped.
Usage and Search IDs come back with the result.

I kept this as one tool. A second firecrawl_search_context would make the agent choose between two overlapping search surfaces. That split is avoidable. Search should mean discovery with optional bounded context. Scrape should stay separate for known URLs.

Why search owns this layer

The Search API already supports search-plus-scrape. The product problem is that the MCP did not expose a search-native way to ask for "some useful page content, capped, focused, and fresh enough for this task."

Search context fields the fields that should be easy for agents
content results, summary, chunks, or markdown.
focus The extraction target. The broad search query and the reading target are often different.
budget maxCharsPerResult and maxCharsTotal are hard caps.
policy Freshness, source policy, sources, categories, country, and location stay explicit.
escape hatch advancedSearchOptions and restricted advancedScrapeOptions. 
{
  "query": "Firecrawl Search API response metadata",
  "content": "chunks",
  "focus": "request fields, response metadata, and defaults",
  "maxResults": 5,
  "maxCharsPerResult": 1200,
  "maxCharsTotal": 5000,
  "freshness": "month",
  "sourcePolicy": {
    "includeDomains": ["docs.firecrawl.dev"]
  },
  "enterprise": ["default"],
  "advancedSearchOptions": {
    "filter": "0"
  }
}

The response stays plain: flattened results, optional content or chunks, metadata, warnings, credits used, Search IDs, and character accounting. Firecrawl should return evidence. The caller should decide what to do with it.

Why MCP owns this layer

The MCP contract is part of the agent's context. A human can read the docs, remember a quirk, and work around a vague field. An agent should not have to rely on training priors or stale documentation to use a tool correctly.

Return shape what happens to the next agent call
Tool returns Immediate result Next query quality
URLs only The agent knows where it might read next. Depends on follow-up scraping and model priors.
Final answer only The current question may be answered. Weak if the evidence never enters caller context.
Bounded evidence The agent gets enough source material to reason. Better, because the next query is conditioned on the web result.

This is the token-efficiency trap. A tool can hide all reading behind another model and return a small answer. The caller's context looks clean, but the caller did not see the evidence. It cannot make the next search sharper because the facts that would have sharpened it never entered the window.

I care about tokens. I care more about saving them at the right layer. Search should compress web evidence into the caller's context, not remove the evidence from the caller altogether.

Executor is the better mental model

The CLI-versus-MCP debate is too shallow. CLIs are useful when the command surface is stable and heavily memorized. MCPs are useful when schema discovery, typed arguments, auth, side effects, and result shape matter. Executor pushes that a step further: one tool surface can discover, describe, and call many typed tools from a code execution context.

This is a real example of the pattern. I can ask Executor to fetch a known web page through Exa, get structured output back, and keep the whole thing inside one TypeScript execution context. 

const page = await tools["mcp.exa.ai"].crawling_exa({
  urls: ["https://docs.firecrawl.dev/features/search"],
  maxCharacters: 12000,
  maxAgeHours: 0
});

return page;

The error path is just as important. I initially called the same tool with url instead of urls. The returned error named the missing schema path, which is exactly the kind of feedback an agent can recover from without guessing. 

const bad = await tools["mcp.exa.ai"].crawling_exa({
  url: "https://alexzhang13.github.io/blog/2025/rlm/",
  maxCharacters: 50000
});

// Returned error:
Input validation error: expected array at path ["urls"]

A shell command can fail too, but the recovery loop is usually less clean: parse stderr, infer which flag was wrong, rerun. In Executor, the schema and the error are part of the same tool context. That is the part I want Firecrawl Search to take seriously. The contract is not documentation after the fact. It is what lets the agent call the tool well.

Implementation choices

One search tool firecrawl_search was upgraded in place. I avoided a second overlapping context-search tool.
content The enum is results, summary, chunks, and markdown.
Hard budgets maxCharsTotal is enforced after extraction. The response cannot silently balloon.
Parity preserved enterprise, filter, source/category inputs, warnings, credits, and Search IDs stay available.
Direct v2 call The MCP uses /v2/search directly while the SDK search path is missing request and response parity.
Deterministic baseline No AI compression yet. Chunk selection should fail in evals before I make it more complex.

Eval plan

The first benchmark should answer a narrow question: does the upgraded contract make agents better at using Firecrawl Search? Correctness matters, but so do context size, source adherence, freshness, tool-call count, latency, credits, and follow-up quality.

4 / 4 live smoke checks passed
9 credits used in smoke validation
1,498ms average smoke latency
Previous MCP Baseline contract: search fields and generic scrape options. to run
Upgraded MCP Content modes, hard budgets, source policy, response accounting. to run
Firecrawl modes results, summary, chunks, and markdown. to run
Exa Reference surface with summaries, highlights, and content controls. to run
Parallel Reference surface focused on LLM-optimized web context. to run

I would run this in three passes: contract checks, Firecrawl mode comparisons, then Exa and Parallel on the same agent tasks. If chunks lose, improve chunks. If summaries win, make summary mode the default. If the contract reduces follow-up calls without hurting answerability, that is the product signal.

The point

Firecrawl already has the hard part: web discovery and extraction. This patch is about making those primitives easier for agents to query correctly.

The MCP should make the intended behavior obvious. Ask for the right context, get bounded evidence back, keep the accounting visible, and make the next call from a better prior. That is the product layer I wanted to add.
