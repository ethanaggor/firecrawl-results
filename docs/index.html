Firecrawl Search needs an agent context layer

I started with the wrong framing: better search quality. The stronger framing is more specific. Firecrawl already has the primitives. The missing layer is a contract that lets agents ask for bounded, useful web context without learning the whole scrape surface first.

EA Ethan Aggor - April 24, 2026
Table of Contents The actual gap The contract What changed Parity issues Live validation Benchmark path

My first instinct was to improve search quality by adding a client-side quality layer. After looking at Firecrawl, Exa, Tavily, and Parallel more closely, I think that was the wrong target. Firecrawl is not primarily missing a reranker. It is missing an agent-native way to shape the context returned by search.

That distinction matters. A reranker assumes the result order is the problem. A context layer assumes the interface is the problem. For AI agents, the practical question is usually not just "which URLs did search find?" It is "how much usable evidence can I get back, at what cost, with which freshness and source constraints?"

The product thesis is simple: Firecrawl Search should expose its search-plus-scrape power as bounded context, not only as shallow results or a broad scrape-shaped options object.

The actual gap

Firecrawl already has serious primitives. Search can discover pages. Scrape can turn pages into clean markdown, summaries, screenshots, links, and structured data. Search can also scrape result pages in the same call through scrapeOptions. That is a strong foundation.

The weak spot is the default MCP-facing shape. Agents either get a lightweight result list, or they need to know enough about the scrape API to request content correctly. That is flexible, but it is not the most queryable contract for agents.

Exa, Tavily, and Parallel make the contrast obvious. Their agent-facing surfaces emphasize content limits, summaries, highlights, chunks, source policy, freshness, and cost/latency tradeoffs. Firecrawl has many of the underlying capabilities, but the MCP tool should package them in a way that matches how agents actually ask for context.

The contract I would expose

I would upgrade firecrawl_search in place. I would not add a second overlapping firecrawl_search_context tool, because that creates split identity for agents. Search should mean discovery plus optional bounded context. Scrape should stay separate for exact known URLs. 

{
  "query": "Firecrawl Search API response metadata",
  "content": "chunks",
  "focus": "request fields, response metadata, and defaults",
  "maxResults": 5,
  "maxCharsPerResult": 1200,
  "maxCharsTotal": 5000,
  "freshness": "month",
  "sourcePolicy": {
    "includeDomains": ["docs.firecrawl.dev"]
  },
  "enterprise": ["default"],
  "advancedSearchOptions": {
    "filter": "0"
  }
}

The important field is content. I prefer that to depth because it describes the output surface directly:

Mode Meaning
results Titles, URLs, descriptions, source, category, and position only.
summary Firecrawl summaries, capped by per-result and total character budgets.
chunks Deterministic extractive chunks selected from markdown using the query and optional focus.
markdown Markdown content, still hard-capped so agents cannot accidentally pull huge context.

The response should be boring on purpose: a flattened result list, optional content or chunks, source metadata, hard budget accounting, warnings, credits used, and Search response IDs. No trust score. No final answer. No next-action recommendation.

What changed in the prototype

The prototype keeps the implementation deterministic. There is no Mercury pass, no semantic compression, and no client-side reranker. Chunks are selected by simple extractive scoring from returned markdown. That gives a baseline we can benchmark before deciding whether an AI compression step is worth it.

It also preserves capabilities that should not have been removed. In particular, enterprise is first-class because Search ZDR is a real Firecrawl capability and was already part of the MCP contract. The implementation also keeps filter as advancedSearchOptions.filter because the live Search endpoint accepts it even though the public docs do not clearly describe it.

What was parity, not product opinion

I separated the product proposal from upstream correctness issues. That matters because a prototype can be opinionated, but parity gaps should be legible on their own.

Node SDK v2 search request parity #3437 Expose full /v2/search response metadata #3438 MCP firecrawl_search missing documented fields #219 /v2/search rejects documented scrapeOptions.zeroDataRetention #3441

The MCP implementation currently calls /v2/search directly for the upgraded search path. That is not meant as a philosophical rejection of the SDK. It is a compatibility decision: the current SDK search path does not yet expose every documented request field or the full response envelope that this MCP tool needs.

What live validation found

The first live smoke was not a full benchmark. It was a net-loss audit: make sure the upgraded tool did not accidentally remove existing Search capabilities while adding the context layer.

4 / 4 live smoke checks passed
9 credits used
1,498ms average latency

The checks covered object-form source/category inputs, advancedSearchOptions.filter and tbs, enterprise: ["default"], Search response IDs, credits used, warnings, and hard character caps.

The interesting failure was ZDR. Firecrawl's Search docs describe combining top-level enterprise: ["zdr"] with scrapeOptions.zeroDataRetention for end-to-end ZDR. In live testing, the endpoint recognized the top-level enterprise field but rejected zeroDataRetention inside Search scrapeOptions. I filed that as a separate docs/API issue because it is independent of the MCP product work.

The benchmark path

I would benchmark this as an agent context problem, not as a standalone SERP ranking problem. The useful questions are relative: does the upgraded MCP contract give an agent enough evidence with fewer tool calls, lower context size, clearer source policy, and predictable cost?

  1. Compare the previous MCP search contract against the upgraded contract.
  2. Compare Firecrawl content modes: results, summary, chunks, and markdown.
  3. Compare Firecrawl against Exa, Tavily, and Parallel on shared agent tasks.

The metrics should include answerability, evidence hit rate, characters returned, credits, latency, source-policy adherence, freshness adherence, and follow-up tool-call count. That is the category where this change should win or fail.

Where I would take this next

I would keep the deterministic baseline first. Once the evals show where summaries or chunks fall short, then it becomes reasonable to ask whether a lightweight model pass should compress chunks, split search query from extraction focus, or choose parameters. Until then, adding AI inside the tool would mostly make the system harder to reason about.

The larger product point is that Firecrawl should not be treated as outside the Exa/Tavily/Parallel conversation. Firecrawl already has the raw web data machinery. The agent-facing search contract is the layer that makes those primitives easy to query, easy to budget, and easy to benchmark.
