# Recommended benchmark methodology for agent web search tools

Use a three-layer benchmark suite rather than a single leaderboard number:

1. **Static factual-answer benchmark for reproducibility.** Use SimpleQA-style short factual questions with known reference answers. Each provider receives the same query or task, retrieved documents are passed into the same answer model and prompt, the model must answer only from retrieved evidence, and responses are graded with the official SimpleQA-style correctness classifier or a fixed equivalent. Tavily's public eval repo runs the full SimpleQA dataset, reformats retrieved documents for `gpt-4.1`, and grades the extracted answer with the official SimpleQA classifier. Tavily's blog says SimpleQA has 4,326 factual questions with ground-truth answers and that GPT-4.1 answered using only retrieved documents. Exa also uses SimpleQA as its factuality/RAG benchmark, with standardized generation over provider results.

2. **Retrieval-quality benchmark that does not depend only on final-answer generation.** Include an IR/relevance layer where the unit being judged is the returned result set or excerpts, not just whether a downstream LLM got the final answer right. Exa uses MSMARCO queries for this purpose: it samples 1,000 MSMARCO queries, sends them to each search API, and uses GPT-4o as judge to score results and snippets on a 1-5 relevance/helpfulness scale. Tavily's public eval repo adds a document-relevance benchmark using QuotientAI and a dynamic dataset, reporting document relevance accuracy separately from SimpleQA accuracy.

3. **Agentic, fresh, multi-step task benchmark.** Add a production-like layer with time-sensitive and multi-hop tasks, because static short-answer datasets miss freshness, ambiguity, and business workflows. Parallel's WISER-Search benchmark is the clearest public pattern: it blends WISER-Fresh and WISER-Atomic, with 40 percent fresh current-event questions and 60 percent hard real-world business questions. The fresh subset has 76 easily verifiable questions generated by o3 pro and tested within 24 hours of generation; the atomic subset has 120 customer-use-case questions. Parallel evaluates accuracy against verified ground truth and reports cost per 1,000 requests, including search API cost plus LLM token cost.

The resulting methodology should report separate scores for:

- **Answer accuracy:** exact or classifier-graded correctness against ground truth, used for SimpleQA and WISER-style tasks.
- **Document relevance / retrieval quality:** judged relevance of returned documents, snippets, or dense excerpts, using a fixed rubric such as Exa's 1-5 MSMARCO judge scale or Tavily's document-relevance evaluator.
- **Grounding discipline:** whether the answer is supported by retrieved documents and whether the model was allowed to use prior knowledge. Tavily's SimpleQA setup explicitly disallows prior knowledge.
- **Freshness performance:** accuracy on tasks whose answer changes or depends on recent web state, with dataset generation time and test time recorded.
- **Agent efficiency:** tool-call count, total latency, input/output tokens, and cost per query or CPM. Parallel's public benchmark treats cost as search API cost plus LLM token cost and emphasizes fewer tool calls and dense excerpts.
- **Source quality and auditability:** URLs used, source diversity, duplicate rate, crawl/fetch failures, and whether source/domain policies were followed.

Task construction should be split deliberately:

- **Held-out static factual set:** SimpleQA or equivalent, preserving comparability and repeatability.
- **Open-web IR sample:** MSMARCO-style query sampling or another query corpus, but judged against the actual public-web results returned by each provider because providers do not share the same index.
- **Dynamic domain-specific dataset:** generated from live web pages for current or domain-specific topics. Tavily docs describe a real-time dataset generator that creates subject-specific search queries, searches the web, generates Q&A pairs from returned sites with a map-reduce workflow, and saves a structured evaluation set for LLM-as-judge evaluation.
- **Freshness set:** generated on a known date, answered within a strict window, and frozen afterward with verified ground truth.
- **Hard agent tasks:** multi-hop business, technical documentation, financial, competitive-intelligence, and navigation-style questions where a real agent would normally search, fetch, compare, and synthesize.

Execution controls should be fixed before running:

- Use the same answering model, prompt, max context, and post-processing path for every provider unless a provider returns a direct answer; if direct answers are allowed, record that path separately as Tavily's repo suggests.
- Run each provider with documented configuration, including search depth, max results, source filters, freshness controls, timeout, and whether returned full text, snippets, or dense excerpts are available.
- Keep provider-specific "best public result" reporting out of the main benchmark. Exa's blog reports the higher of its own result and public SimpleQA results for fairness, but a neutral benchmark should instead rerun all systems under one harness and publish configs.
- Use multiple evaluator models or human spot checks for calibration, especially for LLM-as-judge relevance and business-task correctness.
- Report aggregate results and slices separately: static factuality, retrieval relevance, fresh questions, hard multi-hop tasks, latency, cost, and failure rate. Avoid a single blended score unless the weighting is explicit.

This combines the strongest public patterns: Exa's split between answer factuality and MSMARCO retrieval relevance, Parallel's agent-facing WISER blend with cost and freshness controls, and Tavily's reproducible SimpleQA plus dynamic document-relevance framework.

## Evidence used

- Exa, "Web Search API Evals": SimpleQA setup, MSMARCO sample of 1,000 queries, GPT-4o 1-5 result/snippet judging, and acknowledgement that more real-world benchmarks are needed. https://exa.ai/blog/api-evals
- Parallel, "A state-of-the-art search API purpose-built for agents": WISER-Search blend, 76 WISER-Fresh questions, 120 WISER-Atomic questions, 40/60 distribution, verified-ground-truth LLM evaluation, CPM including API and token cost, and freshness testing dates. https://parallel.ai/blog/search-api-benchmark
- Tavily, "Tavily Evaluation Part 1": SimpleQA has 4,326 questions, GPT-4.1 answers only from retrieved documents, evaluation uses OpenAI correctness prompt, and dynamic/document-relevance evaluation is presented as more production-like. https://www.tavily.com/blog/tavily-evaluation-part-1-tavily-achieves-sota-on-simpleqa-benchmark
- Tavily public eval repo: SimpleQA and Document Relevance benchmarks, `gpt-4.1` post-processing, official SimpleQA classifier, QuotientAI document relevance, dynamic dataset generator, provider configs, sample/range controls, and result output structure. https://github.com/tavily-ai/tavily-search-evals/blob/main/README.md
- Tavily docs, "RAG Evaluation": dynamic web-based RAG dataset generation using domain-specific search query generation, Tavily web search, Q&A pair generation from returned websites, and saving structured eval sets for LLM-as-judge workflows. https://docs.tavily.com/examples/use-cases/web-eval
