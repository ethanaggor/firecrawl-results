# Benchmark Methodology For Agent Web Search Tools

Use a layered agent-trajectory benchmark rather than a single SERP relevance score. The public Exa, Parallel, and Tavily materials point to the same conclusion: search tools for agents should be evaluated on answer correctness, document/result relevance, freshness, cost, latency, context quality, and how many tool calls the agent needs to reach a grounded answer.

## Evidence Base

- Exa API evals: https://exa.ai/blog/api-evals
- Exa eval methodology: https://www.exa.ai/blog/evals-at-exa
- Parallel Search MCP / WISER-Search benchmark: https://parallel.ai/blog/search-api-benchmark
- Tavily search evals repository: https://github.com/tavily-ai/tavily-search-evals
- Tavily search best practices / search depth: https://docs.tavily.com/documentation/best-practices/best-practices-search

## Provider Methodology Comparison

### Exa

Exa uses complementary evaluation frames:

- SimpleQA for downstream factual answer correctness. Exa describes SimpleQA as short factual questions designed to elicit hallucinations, then runs LLMs with web data and scores answers as in the original benchmark.
- MS MARCO-derived open retrieval grading. Exa says MS MARCO is traditionally a fixed-corpus IR benchmark, but fixed labels do not work well for public black-box search APIs with different web indexes. Instead, Exa samples 1,000 MS MARCO queries, sends them to each provider, and has GPT-4o grade returned results and snippets on a 1-5 accuracy/helpfulness scale.
- Open evals over black-box web indexes. Exa's broader eval methodology argues for query-only "open evals": define queries, run each provider against its own index, grade each query-result pair with an LLM, and aggregate relevance scores.
- Multiple grading/aggregation patterns. Exa discusses pointwise, pairwise, and listwise judging, generally defaults to pointwise, reports mean scores for interpretability, evaluates the top 5 results per query, retries failures, uses default API settings apart from result/content settings, reports confidence intervals, and still does manual review.

Useful task patterns from Exa:

- RAG factual QA: SimpleQA question -> search loop -> final answer -> official/LLM grader.
- Result relevance: query -> top N returned documents/snippets -> LLM relevance and quality score.
- Representative open query sets: in-the-wild queries, MS MARCO queries, and hand-crafted hard queries.
- Downstream agent tasks where retrieval is only useful if the agent can complete the larger job, such as coding with docs, real-time web questions, product research, or research discovery.

### Parallel

Parallel's public Search MCP benchmark is explicitly agent-oriented:

- It evaluates a Search MCP server for AI agents rather than standalone search results.
- The benchmark is WISER-Search, a blend of WISER-Fresh and WISER-Atomic.
- WISER-Fresh has 76 current-day questions requiring fresh web data, generated by o3 pro.
- WISER-Atomic has 120 hard real-world business queries based on Parallel customer use cases.
- The blend is 40% WISER-Fresh and 60% WISER-Atomic.
- Tasks cover breaking news, financial data, technical documentation, and competitive intelligence.
- Parallel compares Parallel MCP, Exa MCP/tool calling, and native web search across models including GPT-4.1, o4-mini, o3, and Claude Sonnet 4.
- Reported metrics include accuracy and cost per thousand requests, where cost includes both search API cost and LLM token cost. The benchmark discussion also emphasizes fewer tool calls, denser excerpts, lower context cost, and lower latency.
- Evaluation uses standardized LLM evaluators against verified ground truth answers. Fresh testing was run within 24 hours of WISER-Fresh dataset generation.

Useful task patterns from Parallel:

- Fresh, date-sensitive questions that must be answered against the live web and tested quickly.
- Atomic business questions that require extracting exact facts, doing calculations, or navigating to a specific page.
- Cross-model agent runs where the same search tool is tested as part of an agent workflow.
- Metrics that treat the agent workflow as the unit of evaluation: accuracy, total cost, latency, tool-call count, and context density.

### Tavily

Tavily publishes a public search-evals repository with two main benchmark families:

- SimpleQA: runs the full SimpleQA dataset against each provider; retrieved documents are reformatted for an LLM, using gpt-4.1 in the repository description, to extract a predicted answer; the prediction is graded with the official SimpleQA classifier. If a provider returns direct answers, the README notes that the direct answer can be compared with the SimpleQA ground truth.
- Document relevance: uses QuotientAI to assess retrieved document relevance against a query. The dataset can be generated dynamically with an open-source dynamic eval dataset generator, or run from the provided document_relevance_dynamic_test_set.json. Tavily frames this as useful for domain-specific and real-time topics.
- The Tavily runner exposes simpleqa and document_relevance evaluation types, supports random samples and index ranges, uses gpt-4.1 defaults for post-processing/evaluation/token accounting, and records per-provider outputs and summaries.
- Tavily's docs make search_depth a product surface to evaluate separately. Search depth controls latency, relevance, and content shape: ultra-fast is lowest latency/lower relevance/content summaries; fast is low latency/good relevance/chunks; basic is medium latency/high relevance/content; advanced is higher latency/highest relevance/chunks.

Useful task patterns from Tavily:

- Static factual QA via SimpleQA.
- Dynamic document relevance datasets for current or domain-specific query distributions.
- Mode comparisons where search_depth, max_results, chunks/content, and raw_content settings are recorded as part of the benchmark configuration.

## Recommended Benchmark Design

Build a benchmark with four required layers.

1. Tool and schema footprint

Start every run from MCP tool discovery. Record tool count, search/fetch tools, required/optional fields, schema tokens, description tokens, freshness controls, source/domain controls, content-budget controls, and whether usage metadata is exposed. This measures whether the provider is ergonomic for agents before retrieval even starts.

2. Retrieval relevance

Run open-eval query sets through each provider. Include:

- A MS MARCO-derived query sample for broad IR comparability, but do not use fixed MS MARCO labels as the final truth for public web APIs.
- In-the-wild or production-like queries.
- A hand-crafted hard set covering multi-hop, technical, business, and long-tail queries.

For each query, evaluate the top 5 results by default, with periodic top 10 sensitivity checks. Grade query-result pairs pointwise with an LLM judge on relevance, source authority, factual usefulness, content issues, and confidence. Aggregate mean relevance and confidence intervals; optionally use pairwise/listwise audits on a sample.

3. Downstream answer correctness

Run answerable agent tasks, not only search calls. Include:

- SimpleQA-style factual questions.
- WISER-Fresh-style current questions with verified same-day or recent ground truth.
- WISER-Atomic-style business/technical questions requiring exact facts, calculations, navigation, or source synthesis.

The agent should use the MCP naturally: search, refine, fetch when necessary, then answer. Grade final answers against verified ground truth or official classifiers where available. Record answer accuracy and whether the final answer is supported by visible evidence.

4. Agent workflow efficiency and context quality

For each trajectory, record:

- Search calls, fetch calls, failed calls, retries, and recoverable errors.
- Total latency and latency by tool call.
- Raw input/output tokens and raw MCP response tokens.
- Final evidence tokens and useful-evidence ratio.
- Tokens per useful finding.
- Total cost per query, including search API cost and LLM token cost.
- Whether the provider returned enough dense, inspectable excerpts for the agent to answer without unnecessary scraping.
- Whether source/freshness constraints were obeyed.

## Required Task Mix

Use a fixed benchmark suite with stratified task families:

- Simple factual QA: SimpleQA-style questions for answer correctness.
- Open relevance: MS MARCO-style and in-the-wild queries scored with LLM relevance judges.
- Fresh web facts: WISER-Fresh-style current events and announcements, tested near generation time.
- Atomic business/technical facts: WISER-Atomic-style real-world questions requiring exact extraction or computation.
- Document relevance: Tavily-style dynamic datasets where the question is whether retrieved documents are relevant to the query.
- Product-mode comparisons: run the same tasks across provider modes such as Tavily search_depth or other quality/latency settings.

## Scoring

Report separate scores instead of collapsing everything into one number:

- Answer accuracy: percent correct on verified-answer tasks.
- Retrieval relevance: mean LLM relevance/quality score for top N documents.
- Source quality: share of primary/official sources and domain-policy adherence.
- Grounding: share of final claims supported by returned evidence.
- Freshness: date correctness and compliance with freshness controls.
- Efficiency: tool calls, latency, total cost per query, token use, and retries.
- Agent experience: schema clarity, control expressiveness, output inspectability, follow-up affordance, and error recoverability.

The headline benchmark should rank providers by answer accuracy at a fixed budget, but the report should always show retrieval quality, cost, latency, tool-call count, context size, and evidence grounding separately. This avoids the main failure mode Exa warns about: a scalar downstream score can hide whether the retrieval engine, content extraction, query generation, or agent workflow caused the result.

## Bottom Line

The best methodology is a full agent trajectory benchmark with open retrieval grading and verified-answer tasks. Exa contributes the open-eval and SimpleQA/MS MARCO relevance methodology; Parallel contributes WISER-style fresh and atomic agent tasks with cost/latency/tool-call accounting; Tavily contributes SimpleQA plus document relevance datasets and explicit evaluation of search modes such as search_depth. Together, these imply that agent web search tools should be benchmarked on correctness, document relevance, freshness, context density, latency, cost, source quality, and tool ergonomics in the same run.
